# Game Auto-Updater — C++ / Visual Studio 2022

Two projects in one solution:

| Project | Output | Role |
|---|---|---|
| **AutoUpdateClient** | `AutoUpdateClient.exe` | Runs on player PC, checks + downloads files, launches game |
| **MD5Generator** | `MD5Generator.exe` | Developer tool — scans build folder, writes `AutoUpdate.ini` |

---

## Architecture

```
[AutoUpdate.ini] ← generated by MD5Generator, hosted on your HTTP server
       │
       │  (downloaded by AutoUpdateClient on every launch)
       ▼
 AutoUpdateClient
  1. Download AutoUpdate.ini from http://localhost/Update/AutoUpdate.ini
  2. Parse file list + expected MD5 hashes
  3. For each file:
       - Compute MD5 of local copy
       - If different (or missing) → download from server
  4. Launch Game.exe
```

---

## AutoUpdate.ini format

```ini
[AutoUpdate]
BaseURL=http://localhost/Update

[Files]
Game.exe=d41d8cd98f00b204e9800998ecf8427e
data/maps.pak=5d41402abc4b2a76b9719d911017c592
data/textures.pak=098f6bcd4621d373cade4e832627b4f6
```

- **BaseURL** — where files are downloaded from (can override the built-in default).
- **[Files]** — each line is `relative/path=md5hash` (forward slashes, lowercase hex).

## Build

1. Open `AutoUpdate.sln` in **Visual Studio 2022**.
2. Select `Release | x64` (recommended for distribution).
3. **Build Solution** (`Ctrl+Shift+B`).

### Required: "Desktop development with C++" workload
Must include:
- MSVC v143 compiler
- Windows 10/11 SDK

## Workflow

### Developer side (server update)

```
1. Build your game into a clean folder, e.g. C:\GameBuild\
2. Run:
     MD5Generator.exe "C:\GameBuild" "http://yourserver.com/Update"
3. Upload all files in C:\GameBuild\ (including AutoUpdate.ini) to:
     http://yourserver.com/Update/
```

### Player side (client)

```
1. Player runs AutoUpdateClient.exe
2. It downloads AutoUpdate.ini, checks all files, downloads changed ones
3. Launches Game.exe automatically
```

Edit the three constants at the top of `AutoUpdateClient/main.cpp`:

static const wchar_t* kDefaultBaseURL = L"http://localhost/Update"; // your server URL
static const wchar_t* kIniFileName    = L"AutoUpdate.ini";          // manifest filename

## Antivirus — Why this code is clean

| Factor | This project |
|---|---|
| HTTP library | **WinHTTP** — built-in Windows, fully trusted by AV |
| Hashing | **Windows CryptAPI** — no suspicious code patterns |
| No code injection | Zero use of `VirtualAllocEx`, `WriteProcessMemory`, `CreateRemoteThread` |
| No process hollowing | Files downloaded to disk, not injected into memory |
| Temp-file strategy | Downloads to `.tmp`, then renames — avoids partial overwrites |
| Manifest embedded | Proper UAC + Common Controls v6 manifest — looks legitimate |
| Version resource | `resource.rc` with company/product strings — edit with your info |
| Static CRT (Release) | `RuntimeLibrary=MultiThreaded` — no VCRUNTIME DLL dependency |

### Additional recommendation — Code Signing
The single most effective antivirus measure is **signing the binary** with a code-signing certificate. Even a cheap EV certificate ($100–300/yr) eliminates most AV false positives and Windows SmartScreen warnings. Unsigned `.exe` files that download and write files will always be flagged more aggressively.

## Limitations / Known issues

| Issue | Notes |
|---|---|
| Self-update | AutoUpdateClient cannot update itself while running. Place it outside the update scope, or use a small "launcher stub" pattern. |
| HTTPS | Supported via WinHTTP — set `BaseURL=https://...` and it works automatically. |
| Subdirectories | Created automatically during download. |
| Large files | Progress bar shows per-file and overall progress. |
| Retry | One automatic retry per file on download failure. |
| No rollback | If a download fails mid-update, already-downloaded files stay. Add rollback if required. |
